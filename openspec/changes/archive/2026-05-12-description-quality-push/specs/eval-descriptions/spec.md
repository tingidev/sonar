## ADDED Requirements

### Requirement: Cross-provider judge model

The system SHALL accept a `--judge-model` CLI flag on `sonar eval --descriptions`. When provided, the judge LLM client SHALL be created from `--judge-model` via `create_llm_client()`, independent of the generator model that produced the bundle's descriptions. When `--judge-model` is omitted, the judge SHALL use the model specified by `--model` (preserving backward compatibility).

#### Scenario: Judge uses a different provider than the generator

- **WHEN** `sonar eval --descriptions --judge-model openai/gpt-4o` is invoked against a bundle generated by `anthropic/claude-haiku-4-5-20251001`
- **THEN** the judge LLM calls SHALL route through the OpenAI client
- **AND** the eval report SHALL record both the generator model (from the bundle or `--model`) and the judge model

#### Scenario: Default judge model matches --model

- **WHEN** `sonar eval --descriptions` is invoked without `--judge-model`
- **THEN** the judge SHALL use the same model as `--model`
- **AND** behaviour SHALL be identical to the pre-change implementation

### Requirement: Refined scoring dimensions with integer scale and reasoning

The judge SHALL score three dimensions on an integer scale of 1-5:

- **Accuracy**: correct reflection of what the schema shows -- claims must be supported by column names, types, and structural signals visible to the judge.
- **Specificity**: useful detail, not generic filler -- would this description help distinguish this table from others?
- **Domain inference**: correct identification of the table's domain and use of appropriate terminology

The judge SHALL receive the table's schema (column names, types, nullability, PK flags) and the generated description, but SHALL NOT receive row samples. The scoring measures whether the description is self-contained and useful given only the schema, mirroring what a downstream MCP agent reads before invoking the `sample` tool.

Each dimension SHALL include a reasoning string explaining the score. The judge prompt SHALL define rubric anchors for each integer value.

#### Scenario: Judge returns integer scores with reasoning

- **WHEN** the judge evaluates a table description
- **THEN** the score for each dimension SHALL be an integer in the range 1-5
- **AND** each dimension SHALL include a non-empty reasoning string

#### Scenario: Scores outside valid range are clamped

- **WHEN** the judge returns a score below 1 or above 5 for a dimension
- **THEN** the system SHALL clamp the score to the nearest valid integer (1 or 5)

#### Scenario: Flagging threshold applies to new scale

- **WHEN** a table's description scores below 3 on any dimension
- **THEN** that table SHALL appear in the flagged list with its per-dimension scores and reasoning

### Requirement: Versioned eval artifact output

Each eval run SHALL write a JSON artifact file containing:

- Run timestamp (ISO 8601)
- Generator model identifier
- Judge model identifier
- Prompt version hash (SHA-256 of the system prompt and table prompt template concatenated)
- Per-table scores with reasoning for each dimension
- Aggregate metrics (mean per dimension, flagged count)
- List of tables evaluated (schema.table keys)

The artifact SHALL be written to an `--output` path when specified. When `--output` is not specified, no artifact file SHALL be written (console-only output is preserved).

#### Scenario: Artifact file is written when --output is specified

- **WHEN** `sonar eval --descriptions --output eval-runs/run-001.json` is invoked
- **THEN** a JSON file SHALL be written at `eval-runs/run-001.json`
- **AND** the file SHALL be valid JSON parseable by `json.loads`
- **AND** the file SHALL contain all fields listed above

#### Scenario: No artifact file without --output

- **WHEN** `sonar eval --descriptions` is invoked without `--output`
- **THEN** no artifact file SHALL be written
- **AND** console output SHALL behave as before

#### Scenario: Prompt version hash changes when prompts change

- **WHEN** the system prompt or table prompt template is modified
- **THEN** the prompt version hash in the artifact SHALL differ from the previous run's hash

### Requirement: Fixed-sample evaluation mode

The system SHALL accept a `--sample N` flag on `sonar eval --descriptions`. When provided, the system SHALL evaluate exactly N tables (or all tables if fewer than N have non-null descriptions). Table selection SHALL round-robin across schemas: tables within each schema SHALL be sorted by name, then interleaved by schema in sorted schema order, until N tables are reached. When omitted, all tables with non-null descriptions SHALL be evaluated.

#### Scenario: Sample size is honoured

- **WHEN** `sonar eval --descriptions --sample 10` is invoked against a bundle with 50 described tables
- **THEN** exactly 10 tables SHALL be evaluated

#### Scenario: Sample round-robins across schemas

- **WHEN** `sonar eval --descriptions --sample 10` is invoked against a bundle with 5 schemas each holding 10 described tables
- **THEN** the sample SHALL contain exactly 2 tables from each schema
- **AND** the tables within each schema SHALL be the first 2 by sorted table name

#### Scenario: Sample handles uneven schema sizes

- **WHEN** `sonar eval --descriptions --sample 10` is invoked against a bundle where one schema has 50 described tables and two other schemas have 2 described tables each
- **THEN** the sample SHALL include all 2 tables from each of the smaller schemas
- **AND** the remaining 6 tables SHALL come from the larger schema in sorted order

#### Scenario: Single-schema database degenerates to sorted top-N

- **WHEN** `sonar eval --descriptions --sample 10` is invoked against a bundle with a single schema containing 50 described tables
- **THEN** the sample SHALL be the first 10 tables by sorted table name

#### Scenario: Sample larger than available tables evaluates all

- **WHEN** `sonar eval --descriptions --sample 100` is invoked against a bundle with 20 described tables
- **THEN** all 20 tables SHALL be evaluated

#### Scenario: Deterministic sample across runs

- **WHEN** `sonar eval --descriptions --sample 10` is invoked twice against the same bundle
- **THEN** the same 10 tables SHALL be evaluated in both runs, in the same order
